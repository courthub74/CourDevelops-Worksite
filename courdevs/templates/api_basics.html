RESTful API

Representational State Transfer (REST): is a software architectural style that defines a set of constraints to be used for creating web services. REST-compliant systems, often called RESTful systems, are characterized by how they are stateless and separate the concerns of client and server. Systems that follow the REST paradigm are stateless, meaning that the server does not need to know anything about what state the client is in and vice versa. Because REST systems interact through standard operations on resources, they do not rely on the implementation of interfaces.

If a system violates any of these contraints, it cannot be considered RESTful:

1. Client-Server architecture
2. Statelessnesss
3. Cacheability
4. Layered System
5. Code on Demand (Optional)
6. Uniform Interface




SOAP API

Simple Object Access Protocol (SOAP): These built-in standards do carry a bit more overhead, but can be a deciding factor for organizations that require more comprehensive features in the way of security, transactions, and ACID (Atomicity, Consistency, Isolation, Durability) compliance.


1. SOAP has tighter security
2. Successful/retry logic for reliable messaging functionality
3. SOAP has built-in ACID compliance

Overall SOAP is a Protocol system. REST is an architectural style
